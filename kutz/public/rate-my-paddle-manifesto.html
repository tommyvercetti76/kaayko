The Innovation

Smart Contextual Chips

Unlike any rating system, our chips are generated from real weather data. You physically cannot select a chip that doesn't match conditions. This makes every label trustworthy by construction.

Weather-dependent chips

Only shown when weather data supports them. A paddler at a calm lake on a sunny day will never see "Strong wind" or "Rain."

Strong windwind > 20 kph
Gustygust_factor > 2.0
Cold waterwater_temp < 12°C
Rain / stormprecip_mm > 0.5
Rough waveswave_height > 0.25m
Poor visibilityvis_km < 5
Too hottemp > 32°C
Too coldtemp < 5°C
Big water exposurelake_area > 50 km²

Always-available context chips

Non-weather factors that are always valid. These capture the experience the weather API can't see.

Crowded / boat traffic Launch was difficult Felt unsafe solo Current / flow issues

These always appear as options (after weather chips fill priority slots). Someone on a calm sunny day who rates "2" can still tell us it was the boat traffic or a bad launch ramp.

The Anti-Fake Guarantee

Why max 4 chips? Forces prioritization. "Everything was bad" isn't useful data. "Wind was the main problem" is. Four chips make the paddler choose what mattered most.

Why weather-gated? If the API says wind is 5 kph, we don't offer "Strong wind." A troll can't fabricate complaints that contradict real conditions. Every chip is verifiable.

Engineering

New APIs to Build

POST /api/paddleScore/publicRating NEW
Core endpoint. Receives a public paddle rating with full context. Auto-attaches weather. No auth required.
spotId REQ string — lake identifier matching paddlingSpots
rating REQ integer 1-5
skillLevel REQ enum: beginner | intermediate | advanced | expert
boatType REQ enum: kayak | sup | canoe | rowing | other
groupType REQ enum: solo | partner | group
withDogs OPT boolean
reasonChips OPT string[] (max 4) — validated against weather
windFelt OPT enum: calm | light | moderate | strong | brutal
waveFelt OPT enum: flat | ripple | chop | whitecaps
notes OPT string (280 char max, sanitized)
lat OPT number — GPS latitude
lng OPT number — GPS longitude
deviceFp OPT string — client fingerprint for rate limiting
Server-side auto-attaches: weatherSnapshot (from coordinates + timestamp), gpsVerified (distance check), qualityWeight (coherence score), lake metadata (HydroLAKES)
GET /api/paddleScore/rateConfig/:spotId NEW
Returns everything the rating form needs: lake info, current weather, current predicted score, and eligible smart chips based on conditions.
Returns: { lake, weather, predictedScore, eligibleChips[], chipThresholds, gpsCenter: {lat, lng}, gpsRadiusKm }
POST /api/paddleScore/generateQR NEW
Generates a branded QR code PNG for a specific lake. Returns base64 data URL. Uses existing qrService.js.
Body: { spotId, format: "png" | "svg", size: 512, withLogo: true }
GET /api/paddleScore/publicRatings NEW
Admin-only. Returns aggregated public ratings for monitoring and data quality review. Filterable by lake, date, quality flags.
Auth: requireAdmin. Query: ?spotId=&from=&to=&quality=good|flagged|all
POST /api/paddleScore/syncToTrainer NEW
Admin-only. Exports qualified public ratings to JSONL format compatible with the Paddle Trainer. Applies quality filters.
Auth: requireAdmin. Body: { minQuality: 0.5, since: "2026-05-01" }

Existing APIs to Extend

GET /api/paddleScore EXTEND
Add ?includeChips=true to return eligible smart chips alongside the paddle score. Used by the rating form to know which chips to offer.
GET /api/paddleScore/metrics EXTEND
Add public rating volume, quality distribution, and per-skill-level accuracy breakdowns to the admin metrics endpoint.

What makes v4 different

  • Balanced data across all 6 weather buckets (not 66% cold)
  • Reason tags on every label (22 features activated)
  • Multi-skill training data (beginner through expert)
  • Real paddler ground truth (not hypothetical ratings)
  • Per-spot calibration from public feedback loop
  • Gradient boosted trees (replacing ridge regression)

Model upgrade path
